VisionGate AI

Project: DPWH049 (VisionGate AI | Powered by DP World CARGOES)

1. Why you made this project, what difficuties did you face?
--------------------------------------------------------------------------------
Why I made the project: 
I built VisionGate AI specifically for the DP World National Hackathon to address one of the most significant bottlenecks in port operations: gate container inspections. 
Currently, human triage and manual clipboard logging takes approximately 3-5 minutes per truck. This causes massive truck idling queues, huge localized carbon emissions, and high 
occurrences of human-error logging (driving up cargo liability disputes). VisionGate replaces this with a 14-second fully automated AI-driven visual inspection pipeline using existing 
CCTV cameras. It natively aligns with DP World's "Our World, Our Future" ESG framework.

Difficulties faced:
- Dual-View Image Synchronization: Processing two heavy concurrent image streams (Left/Front and Right/Rear container angles) and injecting them simultaneously into the LLM logic flow 
without causing severe browser/network timeouts or creating duplicate logging artifacts.
- Multimodal Constraints (Document Reasoning vs Vision): The most complex engineering task was injecting unstructured text context extracted from a PDF manifest (via PyPDF2) alongside two 
raw images, and having the AI accurately cross-reference them (e.g. noticing that the PDF declared "No Hazardous Material", but the images displayed a visual Orange Hazmat placard).
- Prompt Engineering JSON Compliance: Enforcing a massive LLM parameter payload to return strict, deterministic JSON variables and exact floating-point spatial bounding box 
arrays (without conversational text bleeding outside the braces) to prevent the UI drawing engines from crashing.

7. For all SDE/ML technolgies you used, why did u use them and not thieir competitior technologies? compare tradeoffs between technologies
--------------------------------------------------------------------------------
- YOLOv8 (over custom CNN): YOLO provides spatially accurate bounding boxes natively. Rather than wasting weeks building/tuning massive ResNet/VGG CNNs from scratch, YOLO runs sub-second 
30-fps inferences fully natively on cheap CPU hardware instead of needing expensive localized cloud GPU servers. 
- EasyOCR (over PaddleOCR): EasyOCR integrates flawlessly within the PyTorch dependency web YOLO already initializes. Implementing PaddleOCR would require adding huge PaddlePaddle framework 
weights which bloats edge-server architecture radically.
- Gemini 2.5 Flash (over GPT-4o / Claude): VisionGate deals with multiple concurrent image layers + extreme string contexts (PDFs, Databases). Gemini Flash offered an unparalleled 
1-million-plus token context window and uniquely low multimodel visual latency constraints, giving the best speed-to-cost processing for high-volume gate traffic.
- Streamlit (over React/Node): When manipulating raw neural networks, Numpy logic matrices, Pandas aggregate graphs, and PIL bounding boxes natively in Python, designing REST APIs to a 
disconnected React Web framework severely inflates development loops. Streamlit compiles Python immediately to a reactive DOM. 
- SQLite (over PostgreSQL): Building a decentralized edge-simulation meant full Postgres relational networking was overkill. SQLite provides rigid, synchronous, lightweight ACID transactional 
integrity directly on the C: drive with zero complicated port deployment configs.
- PyPDF2 (over native Document Vision APIs): Parsing the raw native string arrays directly off digital PDFs is dramatically faster and 100x cheaper than sending an image of a document to an 
OCR vision model, and utterly destroys text LLM hallucinations.

rust damages, causing severe operator reporting errors. 
  Resolution A: I dynamically wrote a schema migration hook in `db_utils.py` to retroactively inject an `inspection_type` schema constraint on thousands of records. I then updated the

  in `st.session_state`. This ensured API logic fired exactly once per container, unlocking blindingly fast UI rendering.

14. What is yolov8n.pt file and what is .pt and how it generated?
--------------------------------------------------------------------------------
The `yolov8n.pt` file contains the pre-trained weights for YOLOv8 Nano, an extremely fast and lightweight object detection ML 
model. The `.pt` extension stands for "PyTorch" and is the standard model file format used to save a PyTorch neural network's state 
dictionary (its trained weights and biases). In this project, it is logically generated by the `download_models.py` script. The script 
uses the Ultralytics framework to download a Container Damage Detection dataset from Roboflow and fine-tunes a base YOLOv8 model on 
it (usually over 25 epochs). After training completes, the network exports and serializes the newly fine-tuned weights into 
the `.pt` file, depositing it into the `/models` directory so it's prepared for offline edge gate inference.

15. What is visiongate.db file and how it is generated?
--------------------------------------------------------------------------------
`visiongate.db` is an embedded SQLite database file that acts as the localized, secure persistence layer for the edge terminal 
simulation. It systematically logs operational gate movements, cargo structural damage records (`container_logs`), and 
logistical warnings (`ship_delays`).
It is auto-generated programmatically on the very first execution of the main application. Under the hood, the system 
imports database initialization logic from `db_utils.py` and rapidly executes `CREATE TABLE IF NOT EXISTS` routines. 
Since SQLite is zero-configuration, its C-engine dynamically builds the `.db` binary mapping strictly onto the root 
folder drive the instant it runs, achieving immediate ACID transactional logging.

--------------------------------------------------------------------------------
Ultralytics is a premier AI company and the creator behind the globally popular open-source YOLO (You Only Look Once) 
computer vision frameworks (specifically YOLOv5 and YOLOv8). In VisionGate, the `ultralytics` Python package 
acts as the core ML dependency powering the localized object detection engine. It heavily abstracts advanced PyTorch

19. What all regex are used, why are they used and where is it used? And how are they used?
--------------------------------------------------------------------------------
Answer:
Regular Expressions (Regex) are mathematically robust string constraint formulas utilized in two main files.
1. **File:** `edge_ml_pipeline.py`
   - **Regex:** `ISO_PATTERN = re.compile(r"[A-Z]{3,4}\s?\d{6,7}")`
   - **Why/How:** Used to securely extract standardized Shipping Container ISO 6346 code markings. EasyOCR natively blind-reads 
   all text on the frame (e.g., random warning stickers or graffiti). The regex ensures only valid ISO patterns 
   (like "MEDU 123456 7") are parsed via `re.findall(ISO_PATTERN, text)` while discarding the rest of the noise.
2. **File:** `app.py`
   - **Regex:** `re.search(r"```(?:json)?\s*([\s\S]*?)```", text)`
   - **Why/How:** Generative AI models (like Gemini) often wrap JSON responses inside Markdown backticks 
   strings (```json ... ```) despite strict explicit instructions not to. This regex is used in both 
   `analyze_container_gemini_dual()` and `analyze_thermal_gemini()` to explicitly slice off the markdown formatting if it
    occurs, leaving only the pure internal JSON data dictionary to successfully bypass python `json.loads()` crash limits safely.

20. What are 25 epochs and batch size 16?
--------------------------------------------------------------------------------
Answer:
These are Critical Machine Learning Hyperparameters defined during training in `download_models.py`.
- **25 Epochs**: An epoch represents one complete forward and backward pass of the *entire* training dataset through the
 neural network. 25 epochs mean the YOLOv8 model views and learns from the dataset completely 25 individual times, iteratively 
 adjusting its internal weights via loss gradients to maximize bounding-box accuracy.
- **Batch Size 16**: Rather than passing the entire dataset into GPU memory at once (which causes system crashes), batching 
slices exactly 16 images simultaneously per step. The model computes the loss error across these 16 images and updates weights 
once per batch. It offers strict stability limits to learning gradients.

21. What is shutil? Where is it used and why and how is it used?
--------------------------------------------------------------------------------
Answer:
`shutil` is a built-in Python standard library used for high-level file system operations natively matching OS shell commands.
- **Where**: It is heavily utilized inside `download_models.py`.
- **Why/How**: When PyTorch finishes the 25 compute epochs, it deeply buries the final `.pt` weights deeply inside a 
dynamically generated artifact path such 
as `runs/detect/train/weights/best.pt`. `shutil.copy(best_model_path, "models/container_damage_yolov8.pt")` 
programmatically intercepts that final output file, copies it explicitly, and physically deposits it cleanly into the 
main `/models` directory path. This ensures that when the live gateway edge system boots, it can systematically discover the 
file structurally without hard-coded obscure directory paths.

23. What all is implemented to handle PDFs in gate inspector pdf attachments, AI chat attachment, and generating pdf reports? How and where all this is implemented?

Answer:
The system manages bidirectional PDF interactions (both reading inward, and generating outward) via two entirely 
separate libraries natively integrated into `app.py`.

**A. Reading PDFs Inward (Cargo Manifests / Documentation)**
- **Library**: `PyPDF2` (specifically `PdfReader`)
- **How it's implemented**: When users upload a PDF file object to Streamlit `st.file_uploader`, the 
`extract_text_from_pdf()` helper script traps the binary blob utilizing `io.BytesIO(pdf_bytes)`. It iterably 
navigates `reader.pages`, dumping physical unicode extracts through `.extract_text()` sequences. 
- **Where it’s used**:
  1. *Gate Inspector Layer*: User attaches a manifest. Extracted textual logic flows straight into the 
  `cargo_doc_text` variable heavily injected into the Gemini Vision context layer prompting it explicitly to verify 
  visual OCR identifiers against the explicitly uploaded document structurally.
  2. *Yard Copilot Layer*: Provides an 'Operational Document' upload pipeline. Drops extraction data into 
  global `st.session_state.copilot_pdf_context`. The Chat AI logically dynamically prepends this context into the 
  overarching System instruction prompt answering any question grounded flawlessly inside the uploaded physical PDF content limits.

**B. Generating PDF Reports Outward (Gate Audit / Compliance Tools)**
- **Library**: `FPDF` (A pure Python PDF generation SDK).
- **How it's implemented**: Implemented strictly utilizing the class structures inside 
`build_audit_pdf_bytes()` and `build_delay_audit_pdf_bytes()`. A custom 
`AuditPDF(FPDF)` instance maps custom overloads resolving dynamic `header()` sequences. 
Code heavily utilizes metric grid commands sequentially:
  - `pdf.add_page()` initializes memory.
  - `pdf.set_font('Helvetica', ...)` structures literal styling.
  - `pdf.cell(w, h, txt, border, ln)` dynamically forces physical column grids rendering

- The pipeline executes SQL via `db_utils` fetching the target row outputs. 
It cascades `for r in logs:` effectively mapping dictionary properties explicitly into 
independent cell blocks statically generating formatting alignments securely.
- Upon completion, `bytes(pdf.output(dest='S'))` writes the generation straight into 
RAM allowing Streamlit API (`st.download_button()`) to seamlessly intercept and push the

24. PyPDF2 reads input PDF but what does this output?
--------------------------------------------------------------------------------
Answer:
When `PyPDF2.PdfReader` ingests the raw binary input PDF bytes via `io.BytesIO()`, it 
extracts the content and outputs a **single, continuous Python string object** containing 
all the plain-text characters found within the document. 

Specifically, the `extract_text_from_pdf` helper function loops through every 
logical page (`reader.pages`), calls the `.extract_text()` method, and 
concatenates it together continuously. The final output is raw, unformatted Unicode 
text (e.g., `text += page.extract_text()`). This pure string variable is 
what is ultimately injected directly into the Gemini LLM Context Window, 
allowing the AI to "read" the document simply as standard text without needing 
complex binary decoders.

25. Explain the working of the `.extract_text()` method. Is it custom written code or is it implicit in PyPDF and just called?
--------------------------------------------------------------------------------
Answer:
The `.extract_text()` method is **implicit** to the third-party `PyPDF2` library. 
We did not write the underlying low-level parsing logic ourselves; we simply install the 
package and call the method as an API provided by the `PdfReader.pages` class.

**Detailed Working Mechanism:**
When our script executes `page.extract_text()`, the PyPDF2 library automatically handles a 
highly complex background decoding process natively:
1. **Stream Execution:** It scans the low-level binary data streams constructing that specific 
PDF page to locate physical PostScript text-drawing operators (such as the `Tj`, `TJ`, or `Tm` 
commands).
2. **Coordinate Matrix Mapping:** Fundamentally, PDFs do not natively "know" what a sentence 
is. A PDF simply stamps individual characters at mathematical X/Y coordinates on a visual 
geometric canvas. PyPDF2 analyzes the horizontal and vertical distances between these placed 
characters to heuristically calculate where "spaces" ( ) or "newline breaks" (`\n`) logically 
belong.
3. **Unicode Translation:** It evaluates the internally embedded font mapping tables 
(like `ToUnicode` dictionaries or `CIDFonts`) to translate the raw proprietary Adobe 
character encoding identities back into standard, human-readable UTF-8 characters.
4. **Return Protocol:** Once the full coordinate map is solved, it packages the decoded 
results and returns them back to our `app.py` execution thread purely as a single continuous 
Python string.
